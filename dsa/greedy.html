Greedy Algorithms

At each step, choose the locally optimal option. Hope that local choices add up to a global optimum. Recognise the pattern — the solution almost always follows one of 5 skeletons.

01 — Core idea

The greedy choice property: A globally optimal solution can be reached by making locally optimal (greedy) choices at each step. You never reconsider a past decision. Works only when the problem has both greedy-choice property AND optimal substructure.

Greedy-choice property
Local best = Global best
The locally optimal choice at each step leads to a globally optimal solution. The key property that makes greedy work.
Optimal substructure
Subproblems are optimal too
An optimal solution contains optimal solutions to its subproblems. Shared with DP — but greedy doesn't explore all options.
When greedy fails
No greedy-choice property
0/1 Knapsack, Longest Common Subsequence. A locally good choice blocks a globally better path. Use DP instead.
Exchange argument
How to prove greedy works
Show that any solution that differs from the greedy solution can be transformed into the greedy solution without getting worse.
Time complexity
Typically O(n log n)
Usually dominated by the sorting step. Greedy processing itself is linear. Heap-based greedy adds O(log n) per operation.
02 — Problem-solving checklist
When you see a new problem, run through these questions before writing any code.
Decision framework
QuestionWhat you're checkingIf yes
Sort / prioritise?Is there a natural ordering (end time, ratio, frequency)?Sort first — likely Sort+Pick pattern

Signs a problem is greedy: "minimum number of", "maximum number of", "can you reach", "arrange to minimise cost", interval/scheduling problems, fractional versions of knapsack-type problems.

red flags — use DP instead

If you need to pick a subset (0/1 choices), if taking one item blocks another better combination, or if you can find a counterexample where the greedy choice gives a suboptimal result.

04 — Pattern 1: Sort + Pick
Sort elements by the right key, then greedily pick in order. The "right key" is the most common interview insight to arrive at.
Skeleton
// sort by end time (activity selection)
Arrays.sort(arr, (a, b) -> a[1] - b[1]);
int lastEnd = Integer.MIN_VALUE;
int count = 0;
 
for (int[] interval : arr) {
if (interval[0] >= lastEnd) {
count++;
lastEnd = interval[1];
}
}
What to sort by
End timeActivity selection, meeting rooms — earliest finish first
Value/weight ratioFractional knapsack — highest ratio first
DeadlineJob scheduling — earliest deadline first
Start timeMerge intervals, room allocation
Profit descJob sequencing with deadlines
Length ascMinimum cost to cut a rod / rope
Activity Selection
Maximum non-overlapping intervals
Sort by end time. Pick the interval that ends earliest and doesn't overlap with the last picked. Classic greedy proof.
Fractional Knapsack
Sort by value/weight ratio
Unlike 0/1 knapsack, you can take fractions. Sort by ratio desc, fill greedily. O(n log n) vs DP's O(nW).
Job Scheduling
Maximise profit with deadlines
Sort by profit desc. For each job, assign to the latest available slot before its deadline. Use a boolean slot array.
05 — Pattern 2: Heap / Priority Queue
When you need to dynamically pick the smallest or largest element as you process the input. Sort + Pick won't work because the "best" choice changes as costs accumulate.
Skeleton — min-heap merge
// connect ropes / huffman
PriorityQueue<Integer> pq =
new PriorityQueue<>();
for (int num : arr) pq.add(num);
 
int totalCost = 0;
while (pq.size() > 1) {
int a = pq.poll();
int b = pq.poll();
int cost = a + b;
totalCost += cost;
pq.add(cost);
}
Skeleton — meeting rooms II
// min-heap of end times
Arrays.sort(intervals, (a,b) -> a[0]-b[0]);
PriorityQueue<Integer> ends =
new PriorityQueue<>();
 
for (int[] i : intervals) {
if (!ends.isEmpty() &&
ends.peek() <= i[0]) {
ends.poll(); // reuse room
}
ends.add(i[1]); // new/reused room
}
return ends.size();
Huffman Encoding
Optimal prefix-free codes
Always merge the two lowest frequency nodes. Min-heap gives O(n log n). Assigns shorter codes to more frequent chars.
Meeting Rooms II
Minimum rooms needed
Sort by start. Min-heap of end times. If earliest-ending room is free, reuse it. Heap size = rooms needed. LC 253.
Connect Ropes
Minimum cost to connect
Always connect the two shortest ropes first. The cost is the sum of lengths. Classic Huffman variant.
06 — Pattern 3: Running Variable
No sorting, no heap. Just iterate and maintain a running variable that tracks the furthest reach, maximum sum, or cumulative state. Reset when the running variable goes negative.
Skeleton — Jump Game
// LC 55 — can you reach the end?
int maxReach = 0;
for (int i = 0; i < nums.length; i++) {
if (i > maxReach) return false;
maxReach = Math.max(
maxReach, i + nums[i]);
}
return true;
Skeleton — Gas Station
// LC 134 — find starting station
int total = 0, tank = 0, start = 0;
for (int i = 0; i < gas.length; i++) {
int diff = gas[i] - cost[i];
total += diff;
tank += diff;
if (tank < 0) {
start = i + 1;
tank = 0;
}
}
return total >= 0 ? start : -1;
Jump Game I & II
LC 55 & 45
Track maxReach. For min jumps (LC 45), track current boundary and next boundary — increment jumps when you reach the boundary.
Gas Station
LC 134 — circular route
If total gas >= total cost, a solution exists. Reset start whenever tank goes negative — the segment before was unviable.
Best Time to Buy Stock II
LC 122 — unlimited trades
Collect every positive daily difference. If tomorrow is higher than today, buy today and sell tomorrow. Pure greedy.
07 — Pattern 4: Frequency / Counting
Count character or task frequencies. Then greedily use the most frequent element first to avoid constraint violations (adjacent same, cooldown, etc.).
Skeleton — max-heap by frequency
// rearrange string / task scheduler
Map<Character, Integer> freq =
new HashMap<>();
for (char c : s.toCharArray())
freq.merge(c, 1, Integer::sum);
 
// max-heap by frequency
PriorityQueue<int[]> pq =
new PriorityQueue<>((a,b)->b[1]-a[1]);
for (var e : freq.entrySet())
pq.offer(new int[]{e.getKey(), e.getValue()});
 
// greedily place most frequent first
Task Scheduler — math trick
// LC 621 — O(n) without simulation
int[] count = new int[26];
for (char c : tasks) count[c-'A']++;
Arrays.sort(count);
int maxFreq = count[25];
int maxCount = 0;
for (int c : count)
if (c == maxFreq) maxCount++;
 
int frames = (maxFreq - 1) * (n + 1) + maxCount;
return Math.max(frames, tasks.length);
Task Scheduler
LC 621 — cooldown constraint
Most frequent task determines the frame structure. Formula: (maxFreq-1)*(n+1) + countOfMaxFreq. Take max with tasks.length.
Rearrange String k Distance
No same char within k positions
Max-heap + queue of size k. Poll most frequent, place it, decrement count, re-add to heap after k steps.
Reorganize String
LC 767 — no adjacent same
Special case where k=2. Check max frequency <= (n+1)/2. Fill even indices first with most frequent character.
08 — Pattern 5: Two Pointers
Sort first. Then use two pointers to greedily match or pair elements. The key insight is always trying to satisfy the hardest constraint with the best available match.
Skeleton — assign cookies (LC 455)
// match smallest child with smallest cookie
Arrays.sort(children);
Arrays.sort(cookies);
int child = 0, cookie = 0;
while (child < children.length &&
cookie < cookies.length) {
if (cookies[cookie] >= children[child])
child++; // satisfied
cookie++; // try next cookie
}
return child;
Skeleton — boats (LC 881)
// pair heaviest with lightest
Arrays.sort(people);
int i = 0, j = people.length - 1;
int boats = 0;
while (i <= j) {
if (people[i] + people[j] <= limit)
i++; // lightest fits too
j--; // heaviest always goes
boats++;
}
return boats;
Assign Cookies
LC 455 — smallest satisfying match
Sort both. Match each child with the smallest cookie that satisfies them. Move cookie pointer regardless; move child only on success.
Boats to Save People
LC 881 — pair heavy + light
Sort. Heaviest person must go on a boat. Try pairing with lightest. If they fit, both go. Otherwise heaviest goes alone.
Two City Scheduling
LC 1029 — send n to each city
Sort by cost difference (costA - costB). Send first n to city A (greedy: lowest extra cost), rest to city B.

10 — Greedy vs Dynamic Programming
The hardest skill is knowing when to use greedy and when to switch to DP. Both require optimal substructure — the difference is whether greedy choices are always safe.
Use Greedy when
Fractional choicesYou can take partial items (fractional knapsack)
Provably safeExchange argument proves greedy choice never costs you
Locally = globallyBest local choice is always part of a global best
IntervalsMost interval scheduling problems have greedy solutions
SimplerO(n log n) vs DP's O(n²) or O(nW)
Use DP when
0/1 choicesTake or leave each item (0/1 knapsack)
Counterexample existsYou can find a case where greedy fails
Paths/sequencesLongest common subsequence, edit distance
Multiple statesDecision depends on previous decisions
Overlapping subproblemsSame subproblem solved many times
Fractional vs 0/1 Knapsack — the canonical example
The same problem, two completely different algorithms. Understanding why one is greedy and one needs DP is the key interview insight.
Coin Change — greedy fails!
LC 322 — use DP not greedy
With coins [1,3,4] and target 6: greedy picks 4+1+1=3 coins but optimal is 3+3=2 coins. Classic counterexample.
Interval problems — greedy vs DP
Which interval problems need DP?
Max non-overlapping = greedy. Weighted interval scheduling (each interval has a value) = DP. The weight is the deciding factor.
