01 — Core concepts & mental model
Before syntax, understand the mental model. GitLab CI and GitHub Actions share the same fundamental idea but use completely different terminology for the same things.
⬡ GitLab CI — terminology
PipelineThe top-level automation run. Triggered by push, MR, schedule, API, or manual.
StageA named phase. Jobs in the same stage run in parallel. Stages run sequentially.
JobThe atomic unit. A named block with a script, runs on a runner.
RunnerThe agent that executes jobs. Registered globally, per-group, or per-project.
ExecutorHow the runner runs jobs: shell, docker, kubernetes, etc.
.gitlab-ci.ymlThe single pipeline definition file at repo root.
includePull in external YAML files to compose the pipeline.
rules / onlyControl when a job runs.
artifactsFiles passed between jobs in a pipeline.
cacheFiles persisted across pipeline runs (e.g. node_modules).
⬡ GitHub Actions — terminology
WorkflowEquivalent to Pipeline. Defined in .github/workflows/*.yml
JobEquivalent to Stage+Job combined. Jobs run in parallel by default.
StepAtomic unit inside a job. Equivalent to a single command in GitLab's script.
RunnerThe agent. GitHub-hosted (ubuntu-latest) or self-hosted.
ActionA reusable step from the marketplace or local. No GitLab equivalent (closest: include).
.github/workflows/Directory of workflow YAML files. Multiple files = multiple workflows.
Reusable workflowEquivalent to GitLab's include + template jobs.
ifConditional expression to control job/step execution.
artifactsUses actions/upload-artifact and actions/download-artifact.
cacheUses actions/cache action.
Triggers
What starts a pipeline
Push, MR/PR, schedule, manual, API, upstream pipeline, and more.
Execution order
Parallel vs sequential
GitLab: parallel within stage, sequential between stages. GH: parallel by default, use needs: for ordering.
Conditions
When jobs run
GitLab rules vs only/except. GitHub Actions if expressions. Both support branch, tag, file path filtering.
Contexts & metadata
Runtime information
CI_COMMIT_SHA, CI_PIPELINE_ID (GL) vs github.sha, github.run_id (GH). Access branch, tag, actor, ref.
02 — Pipeline anatomy
The full structure of a pipeline file. Every key explained with its GitLab and GitHub Actions equivalent.
⬡ GitLab CI — .gitlab-ci.yml
# global defaults
default:
image: python:3.13-slim
before_script:
- pip install -r requirements.txt
retry: 2
timeout: 30 minutes
 
# define stages (order matters)
stages:
- lint
- test
- build
- deploy
 
# a job
unit-test:
stage: test
script:
- pytest tests/unit
rules:
- if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
- if: '$CI_COMMIT_BRANCH == "main"'
artifacts:
reports:
junit: build/reports/unit-tests.xml
when: always
tags:
- k8s-runner
⬡ GitHub Actions — .github/workflows/ci.yml
# workflow metadata
name: CI
 
# what triggers this workflow
on:
push:
branches: [main]
pull_request:
branches: [main]
 
# global env available to all jobs
env:
PYTHON_VERSION: '3.13'
 
jobs:
unit-test:
runs-on: ubuntu-latest
timeout-minutes: 30
steps:
- uses: actions/checkout@v4
- uses: actions/setup-python@v5
with:
python-version: ${{ env.PYTHON_VERSION }}
- run: pip install -r requirements.txt
- run: pytest tests/unit
- uses: actions/upload-artifact@v4
if: always()
with:
name: test-results
path: build/reports/
needs / depends-on
Job dependencies
GitLab needs: skips stage ordering. GH needs: makes jobs sequential. Both pass artifacts between jobs.
Matrix builds
Run across multiple versions
Test against multiple Python/Node versions, OS, or any variable in one job definition.
Manual & scheduled
Human-gated & cron jobs
Block deployment behind a manual approval step. Run nightly builds on a cron schedule.
03 — Runners & executors
Runners are the agents that pick up and execute jobs. Understanding runner types and executors is critical for debugging why jobs behave differently in CI vs locally.
⬡ GitLab Runners
Shared runnerProvided by GitLab.com. Available to all projects. Rate limited.
Group runnerAvailable to all projects in a group. Register once, use everywhere.
Project runnerDedicated to one project. Most control.
tags:Target specific runners. Job only runs on runner with matching tags.
Executors
shellRuns directly on runner host. Fast, no isolation.
dockerEach job in a fresh container. Most common.
docker+machineAuto-scales cloud VMs per job. GitLab SaaS.
kubernetesEach job in a k8s pod. Best for cloud-native.
instanceNew GitLab runner — nesting support for DinD.
⬡ GitHub Actions Runners
GitHub-hostedubuntu-latest, windows-latest, macos-latest. Fresh VM per job. 2 vCPU, 7GB RAM (free tier).
Self-hostedYour own infrastructure. Register with a token. Persistent or ephemeral.
Larger runnersPaid. Up to 64 vCPU, 256GB RAM. GPU runners available.
runs-on:Selects runner. Can be a label array for self-hosted targeting.
Self-hosted setup
# label your runner
runs-on: [self-hosted, linux, x64, gpu]
 
# container jobs on self-hosted
container:
image: python:3.13-slim
Docker-in-Docker (DinD)
Building images in CI
Running docker build inside a CI container requires special configuration. Two approaches: DinD and socket binding.
Runner configuration
config.toml & best practices
Concurrency, resource limits, pull policies, cache dir, and security hardening for production runners.
04 — Variables & secrets
Variables are the backbone of flexible, secure pipelines. Understand scope, precedence, masking, and how to securely inject secrets without leaking them.
⬡ GitLab CI — variables
# pipeline-level
variables:
APP_ENV: production
IMAGE_TAG: $CI_COMMIT_SHORT_SHA
 
# job-level (overrides pipeline)
deploy:
variables:
APP_ENV: staging
 
# predefined variables (always available)
$CI_COMMIT_SHA # full commit hash
$CI_COMMIT_SHORT_SHA # short hash
$CI_COMMIT_BRANCH # branch name
$CI_COMMIT_TAG # tag name
$CI_PIPELINE_ID # unique pipeline ID
$CI_JOB_ID # unique job ID
$CI_PROJECT_PATH # namespace/project
$CI_REGISTRY_IMAGE # project container registry
$CI_ENVIRONMENT_NAME # deployment environment
$GITLAB_USER_LOGIN # who triggered
⬡ GitHub Actions — variables
# workflow-level env
env:
APP_ENV: production
IMAGE_TAG: ${{ github.sha }}
 
# job-level env (overrides workflow)
jobs:
deploy:
env:
APP_ENV: staging
 
# github context (always available)
${{ github.sha }} # full commit SHA
${{ github.ref_name }} # branch or tag name
${{ github.run_id }} # unique workflow run ID
${{ github.actor }} # who triggered
${{ github.repository }} # owner/repo
${{ github.event_name }} # push, pull_request, etc.
${{ github.event.inputs.* }} # workflow_dispatch inputs
${{ runner.os }} # Linux, Windows, macOS
Secrets
Masked & protected variables
GitLab masked/protected vars vs GitHub encrypted secrets. Scope, inheritance, and how to avoid leaking secrets in logs.
Variable precedence
Which value wins?
Full precedence chain for both platforms. Job-level always beats pipeline-level. Secrets beat regular vars.
Dynamic variables
Pass vars between jobs
GitLab dotenv artifacts. GitHub GITHUB_OUTPUT / GITHUB_ENV. Set computed values in one job, use in another.
05 — Cache & artifacts
Cache and artifacts are fundamentally different things. Cache speeds up jobs by reusing files across runs. Artifacts pass files between jobs in the same pipeline. Getting this wrong is one of the most common CI mistakes.

Rule of thumb: Cache = speed optimisation (node_modules, pip cache, Maven .m2). Artifacts = pipeline data flow (built binaries, test reports, coverage XMLs). Never use artifacts where cache is appropriate and vice versa.

⬡ GitLab CI — cache
cache:
key:
files:
- requirements.txt # key changes when file changes
prefix: $CI_JOB_NAME
paths:
- .venv/
- .cache/pip/
policy: pull-push # pull | push | pull-push
when: on_success
⬡ GitHub Actions — cache
- uses: actions/cache@v4
with:
path: |
~/.cache/pip
.venv
key: pip-${{ runner.os }}-${{ hashFiles('requirements.txt') }}
restore-keys: |
pip-${{ runner.os }}-
pip-
⬡ GitLab CI — artifacts
build:
stage: build
script: make build
artifacts:
paths:
- dist/
expire_in: 1 week
when: always # on_success | on_failure | always
reports:
junit: build/reports/tests.xml
coverage_report:
coverage_format: cobertura
path: build/reports/coverage.xml
⬡ GitHub Actions — artifacts
# upload
- uses: actions/upload-artifact@v4
if: always()
with:
name: dist-${{ github.sha }}
path: dist/
retention-days: 7
 
# download in another job
- uses: actions/download-artifact@v4
with:
name: dist-${{ github.sha }}
path: dist/
Cache key strategy
Designing effective cache keys
Too broad = stale cache. Too narrow = no hits. The right key invalidates when dependencies change, not on every commit.
Test & coverage reports
Built-in report integration
GitLab natively renders JUnit, Cobertura, SAST, DAST reports in MR UI. GitHub uses third-party actions for this.
06 — Docker integration
Building, tagging, scanning, and pushing Docker images is one of the most common CI/CD tasks. The approach differs significantly between GitLab and GitHub Actions.
⬡ GitLab CI — Docker build & push
build-image:
stage: build
image: docker:26
services:
- docker:26-dind
variables:
DOCKER_TLS_CERTDIR: "/certs"
before_script:
- docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
script:
- docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
- docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
rules:
- if: '$CI_COMMIT_BRANCH == "main"'
⬡ GitHub Actions — Docker build & push
build-image:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- uses: docker/setup-buildx-action@v3
- uses: docker/login-action@v3
with:
registry: ghcr.io
username: ${{ github.actor }}
password: ${{ secrets.GITHUB_TOKEN }}
- uses: docker/build-push-action@v5
with:
push: true
tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
cache-from: type=gha
cache-to: type=gha,mode=max
Kaniko
Daemonless image builds
Build Docker images inside Kubernetes without Docker daemon or privileged mode. Safer than DinD for k8s runners.
BuildKit cache
Layer cache in CI
cache-from/cache-to with registry or GitHub Actions Cache backend. Dramatically speeds up builds.
Image scanning in CI
Trivy, Grype, Docker Scout
Scan images after build, fail pipeline on HIGH/CRITICAL CVEs. Integrate with MR comments or PR checks.
07 — Reusable components
Reusability is what separates maintainable CI from spaghetti YAML. Both platforms have mature mechanisms — but they work very differently.
⬡ GitLab CI — include & templates
# include remote templates
include:
- project: 'ska-telescope/templates-repository'
file: 'gitlab-ci/includes/python.gitlab-ci.yml'
- template: 'Security/SAST.gitlab-ci.yml'
- local: '.gitlab/ci/deploy.yml'
- remote: 'https://example.com/ci/template.yml'
 
# hidden job (template)
.base-python:
image: python:3.13-slim
before_script:
- pip install -r requirements.txt
 
# extend a template
lint:
extends: .base-python
script: flake8 src/
 
# YAML anchors
.env-vars: &env-vars
APP_ENV: production
job:
variables:
<<: *env-vars
⬡ GitHub Actions — reusable workflows & actions
# call a reusable workflow
jobs:
call-deploy:
uses: org/repo/.github/workflows/deploy.yml@main
with:
environment: production
secrets: inherit
 
# reusable workflow definition
on:
workflow_call:
inputs:
environment:
type: string
required: true
secrets:
DEPLOY_KEY:
required: true
 
# composite action (local)
# .github/actions/setup-python/action.yml
runs:
using: composite
steps:
- uses: actions/setup-python@v5
- run: pip install -r requirements.txt
shell: bash
extends vs anchors
GitLab inheritance deep dive
extends does deep merge. YAML anchors do shallow merge. Understanding the difference prevents subtle bugs in inherited jobs.
GitHub Marketplace actions
Third-party actions
actions/checkout, docker/build-push-action, and thousands more. Pinning by SHA vs tag — supply chain security implications.
08 — Deployment patterns
How to safely ship to production using CI/CD. Environments, approvals, rollback, and environment-specific configuration.
⬡ GitLab CI — environments & deployments
deploy-staging:
stage: deploy
script:
- helm upgrade --install app ./chart
environment:
name: staging
url: https://staging.example.com
rules:
- if: '$CI_COMMIT_BRANCH == "main"'
 
deploy-prod:
stage: deploy
script:
- helm upgrade --install app ./chart
environment:
name: production
when: manual # human approval required
rules:
- if: '$CI_COMMIT_TAG'
⬡ GitHub Actions — environments & approvals
deploy-staging:
runs-on: ubuntu-latest
environment:
name: staging
url: https://staging.example.com
steps:
- run: helm upgrade --install app ./chart
 
deploy-prod:
runs-on: ubuntu-latest
environment:
name: production # requires reviewer in GH settings
needs: [deploy-staging]
if: startsWith(github.ref, 'refs/tags/')
steps:
- run: helm upgrade --install app ./chart
env:
KUBECONFIG: ${{ secrets.KUBECONFIG }}
GitOps pattern
CI builds, CD reconciles
CI pushes an image tag update to a GitOps repo. ArgoCD or Flux reconciles the cluster state. Decouples build from deploy.
Rollback strategies
When things go wrong
Helm rollback, kubectl rollout undo, re-running a previous pipeline, feature flags. Know which to use when.
Release automation
Tags, changelogs, releases
Semantic versioning, automatic CHANGELOG generation, creating GitHub/GitLab releases from CI on tag push.
09 — Real-world patterns
Common pipeline patterns you'll encounter and implement. Each is a proven solution to a recurring CI/CD problem.
Monorepo pipelines
Only build what changed
Use changed file paths to trigger only the affected service pipelines. Avoid rebuilding everything on every commit.
Test parallelisation
Split tests across jobs
GitLab CI_NODE_INDEX / CI_NODE_TOTAL. GitHub matrix strategy. Reduce test time from 20 min to 5 min.
Image promotion
Build once, deploy many
Build and test the same image artifact. Promote (retag) it through staging → prod. Never rebuild for production.
Skip CI & pipeline control
Controlling when pipelines run
[skip ci] in commit message. rules: changes: for path-based triggers. Interruptible jobs. Pipeline efficiency.
OIDC / keyless auth
No long-lived secrets
GitLab and GitHub both support OIDC tokens to authenticate to AWS, GCP, Azure, Vault without storing credentials.
Notifications
Slack, email, webhooks
Notify on failure, success, or deployment. GitLab integrations vs GitHub Actions Slack action.

Full terminology mapping between GitLab CI and GitHub Actions, plus key concepts defined precisely.
