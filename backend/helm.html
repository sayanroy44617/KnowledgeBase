01 — Core concepts
Helm is the package manager for Kubernetes. It bundles Kubernetes manifests into reusable, versioned packages called Charts.
Chart
Package of K8s manifests
A directory of templates + values that render into Kubernetes YAML. Versioned and shareable.
Release
An installed chart instance
Each helm install creates a release. Same chart can be installed multiple times with different names.
Repository
Chart registry
A server hosting chart packages. Helm Hub, Artifact Hub, or private (Harbor, Nexus, S3, OCI).
Revision
Release history
Every install/upgrade/rollback creates a new revision. Stored as Secrets in the namespace. Enables rollback.

Helm 3 vs Helm 2: Helm 3 removed Tiller (the server-side component). All operations now happen client-side using your kubeconfig. Release state is stored as Secrets in the same namespace as the release — no cluster-wide permissions needed.

Helm 3 changes
What changed from v2
No Tiller, namespaced releases, JSON Schema validation, OCI support, improved security model.
Values
Configuration injection
Default values in values.yaml. Override at install/upgrade time. Multiple --values files, --set flags.
Go templating
Dynamic manifest generation
Helm uses Go's text/template engine. Sprig library functions available. Renders values into YAML.

The commands you'll use every day. Grouped by workflow stage.

helm list -A # all namespaces

helm create mychart

helm template myapp ./mychart -f prod.yaml
helm install --dry-run --debug myapp ./mychart

upgrade flags
Critical upgrade options
--atomic, --wait, --timeout, --force, --cleanup-on-fail. Know what each does before using in production.
--set syntax
Overriding values inline
Nested keys, arrays, booleans, escaping. --set vs --set-string vs --set-json vs --values.
helm-diff plugin
Preview changes before upgrade
helm diff upgrade shows exactly what will change. Essential for safe production deployments.
03 — Chart structure
Every Helm chart follows a standard directory layout. Understanding each file's purpose is fundamental.
Directory layout
mychart/
Chart.yaml # chart metadata
values.yaml # default values
values.schema.json # values validation
.helmignore # like .gitignore
charts/ # sub-charts (dependencies)
templates/
deployment.yaml
service.yaml
ingress.yaml
_helpers.tpl # named templates
NOTES.txt # post-install message
tests/ # helm test pods
Chart.yaml
apiVersion: v2
name: mychart
description: A Helm chart for my app
type: application # or library
version: 0.1.0 # chart version
appVersion: "1.0.0" # app version
dependencies:
- name: postgresql
version: "^13.0.0"
repository: https://charts.bitnami.com/bitnami
condition: postgresql.enabled
Dependencies
Sub-charts & umbrella charts
helm dependency update fetches sub-charts. Condition flags enable/disable them. Pass values via alias.
Library charts
Shared template logic
type: library charts share named templates across multiple charts. Can't be installed directly.
values.schema.json
Validate user input
JSON Schema validates values at install/upgrade time. Catch misconfigurations before they reach the cluster.
04 — Templating
Helm templates use Go's text/template engine with the Sprig library. Mastering the syntax unlocks powerful, reusable chart logic.
Template syntax basics
# access values
image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
 
# release metadata
name: {{ .Release.Name }}
namespace: {{ .Release.Namespace }}
isUpgrade: {{ .Release.IsUpgrade }}
 
# chart metadata
version: {{ .Chart.Version }}
appVersion: {{ .Chart.AppVersion }}
 
# whitespace control
{{- .Values.name -}} # trim whitespace
Control flow
# if/else
{{- if .Values.ingress.enabled }}
# ingress resource here
{{- else if .Values.ingress.tls }}
# tls config
{{- end }}
 
# range (loop)
{{- range .Values.ingress.hosts }}
- host: {{ .host }}
{{- end }}
 
# with (scope change)
{{- with .Values.resources }}
resources: {{- toYaml . | nindent 2 }}
{{- end }}
_helpers.tpl
Named templates & partials
Define reusable template snippets. Use define/template/include. The difference between template and include.
Sprig functions
Essential built-in functions
toYaml, nindent, default, required, quote, trim, upper, lower, printf, tpl, lookup and more.
Common mistakes
Whitespace, indent, toYaml
Wrong nindent, missing quotes, YAML type coercion, boolean/null issues. The bugs that bite everyone.
tpl function
Render values as templates
The tpl function allows values themselves to contain template expressions. Powerful but use carefully — values can reference other values.
05 — Values & overrides
Values flow from multiple sources with a strict precedence order. Understanding this prevents configuration confusion in production.
Values precedence (highest → lowest)
# 1. --set flags (highest)
helm upgrade app ./chart --set replicaCount=3
 
# 2. --set-file
helm upgrade app ./chart --set-file config=./config.txt
 
# 3. -f / --values files (last wins)
helm upgrade app ./chart -f base.yaml -f prod.yaml
 
# 4. values.yaml in chart (lowest)
values.yaml design
Structuring default values
Sensible defaults, required vs optional, using default and required functions, nested vs flat structure.
Global values
Share values across sub-charts
.Values.global is passed to all sub-charts in an umbrella chart. The only cross-chart value sharing mechanism.
--set syntax reference
# nested key
--set image.tag=v1.2.3
 
# array item
--set ingress.hosts[0].host=example.com
 
# array with comma
--set "service.ports={80,443}"
 
# force string (avoid type coercion)
--set-string port="8080"
 
# boolean
--set ingress.enabled=true
 
# null (unset a value)
--set annotations=null
 
# escape dots in key names
--set "annotations.kubernetes\.io/ingress\.class=nginx"
06 — Hooks
Hooks let you intervene at specific points in the release lifecycle — before/after install, upgrade, rollback, and delete.
Hook types
Lifecycle intervention points
pre-install, post-install, pre-upgrade, post-upgrade, pre-rollback, post-rollback, pre-delete, post-delete, test.
Hook weights & policies
Ordering & cleanup
helm.sh/hook-weight controls order. helm.sh/hook-delete-policy controls when the hook pod is cleaned up.
Common use cases
DB migrations, config seeding
Run database migrations pre-upgrade. Seed initial data post-install. Backup before delete.
Hook example — DB migration
apiVersion: batch/v1
kind: Job
metadata:
name: {{ .Release.Name }}-migrate
annotations:
"helm.sh/hook": pre-upgrade,pre-install
"helm.sh/hook-weight": "-5"
"helm.sh/hook-delete-policy": hook-succeeded
spec:
template:
spec:
restartPolicy: Never
containers:
- name: migrate
image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
command: ["python", "manage.py", "migrate"]
07 — Repositories & OCI
Helm 3 supports both traditional HTTP chart repositories and OCI (container registry) based distribution.
OCI registries
Charts as OCI artifacts
Push/pull charts to/from Harbor, ECR, GCR, GHCR. No separate chart server needed. Unified image + chart registry.
Private repositories
Auth & access control
helm repo add with --username/--password. OCI login with helm registry login. CI/CD credential management.
# traditional HTTP repo
helm repo add myrepo https://charts.example.com
helm repo add myrepo https://charts.example.com --username user --password pass
helm install myapp myrepo/mychart --version 1.2.3
 
# OCI registry
helm registry login harbor.example.com --username user --password pass
helm push mychart-0.1.0.tgz oci://harbor.example.com/charts
helm install myapp oci://harbor.example.com/charts/mychart --version 0.1.0
helm pull oci://harbor.example.com/charts/mychart --version 0.1.0 --untar
08 — Deployment patterns
How Helm fits into a real deployment workflow with CI/CD, GitOps, and multiple environments.
Multi-environment
Per-environment values
Base values.yaml + environment-specific overrides (values-staging.yaml, values-prod.yaml). One chart, many configs.
Atomic upgrades
Auto-rollback on failure
--atomic waits for rollout and auto-rolls back if it fails. --wait without --atomic leaves you in a failed state.
GitOps with Helm
ArgoCD & Flux HelmRelease
ArgoCD Application with Helm source. Flux HelmRelease CRD. Declarative chart management in Git.
Helm in CI/CD
GitLab CI / GitHub Actions
Best practices for helm upgrade --install in pipelines. Kubeconfig injection, namespace management, version pinning.
Rollback
helm rollback deep dive
helm rollback reruns the previous release's manifests. Doesn't rollback PVCs, database migrations, or ConfigMaps marked with keep policy.
09 — Debugging
When Helm misbehaves, these are your tools. Most issues come down to template rendering errors or value type problems.
--dry-run --debug
Render without applying
Renders templates and shows what would be sent to Kubernetes. Best first debugging step.
helm template
Local rendering
Renders without a cluster connection. Useful in CI to validate chart output. --show-only for a single template.
Failed releases
Stuck in failed state
helm history to see status. helm rollback or helm uninstall to recover. Pending-upgrade state fix.
# debug workflow
helm template myapp ./chart -f prod.yaml # render locally
helm install --dry-run --debug myapp ./chart # render + validate against cluster
helm lint ./chart --strict # lint for issues
helm get manifest myapp # what's actually deployed
helm get values myapp --all # all computed values
 
# fix stuck pending-upgrade
helm history myapp # see the failed revision
helm rollback myapp 0 # rollback to last good revision
 
# force delete a broken release
helm uninstall myapp --no-hooks # skip hooks if they're broken
10 — Patterns & interview topics
Common interview questions and real-world Helm patterns you should know.
Umbrella chart pattern
Deploy a full stack as one chart
A parent chart with sub-chart dependencies (app + postgres + redis). One helm install deploys everything. Trade-offs.
Helm vs Kustomize
When to use which
Helm = packages with logic and versioning. Kustomize = patch-based overlays on existing YAML. Not mutually exclusive.
helm test
Release verification tests
Run test pods post-install to verify the release works. Annotated with helm.sh/hook: test.
Resource policies
Keep resources on uninstall
helm.sh/resource-policy: keep prevents Helm from deleting a resource on uninstall. Use for PVCs you want to preserve.
NOTES.txt
Post-install instructions
Displayed after helm install/upgrade. Use to show connection strings, next steps, access URLs. Also a template file.
lookup function
Query the cluster from templates
Look up existing Kubernetes resources from within a template. Get a Secret value, check if a resource exists.
