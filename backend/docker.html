01 — Docker Architecture
Docker uses a client-server architecture. The CLI talks to the daemon via a REST API over a Unix socket or TCP. The daemon manages images, containers, networks, and volumes. containerd is the actual container runtime (OCI-compliant) that Docker delegates to.
02 — Images & Layers
Union filesystem
Layered image structure
Each instruction in a Dockerfile creates a read-only layer. Layers are cached and shared across images.
Build cache
Layer caching
Docker reuses unchanged layers. Order instructions from least to most frequently changed to maximise cache hits.
Multi-stage builds
Smaller production images
Use multiple FROM statements. Build in one stage, copy only artifacts to the final stage. Drastically reduces image size.
Image layer flow
Base image
→

→
COPY src
→
RUN pip install
→
Container layer (R/W)

03 — Dockerfile
Key instructions
FROMRUNCOPY ADDENVARG EXPOSEWORKDIRUSER
Each instruction adds a layer. Only RUN, COPY, ADD create layers — others are metadata.
ENTRYPOINT vs CMD
Container startup
ENTRYPOINT = the executable. CMD = default args. Together they compose the startup command.
Best practices
Production-ready images
Use specific base tags, non-root user, .dockerignore, minimal base images, combine RUN commands.
Multi-stage example — Python FastAPI
# build stage
FROM python:3.13-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
 
# final stage
FROM python:3.13-slim
WORKDIR /app
COPY --from=builder /usr/local/lib/python3.13 /usr/local/lib/python3.13
COPY src/ .
USER 1000:1000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]
04 — Containers
Lifecycle
Created → Running → Stopped
Containers have a writable layer on top of the image. Data is lost when container is removed unless using volumes.
Runtime flags
docker run options
-d detached, -p port map, -v volume mount, -e env var, --name, --network, --restart policies.
Debugging
Inspect & troubleshoot
exec into running containers, inspect for full config, stats for resource usage, events for daemon activity.
# run & manage
docker run -d -p 8080:80 --name web nginx
docker run --rm -it python:3.13 bash
docker ps -a
docker stop web && docker rm web
docker restart web
# inspect & debug
docker logs web -f --tail=100
docker exec -it web bash
docker inspect web
docker stats
docker top web
05 — Networking
bridge
Default network driver
Containers on the same bridge can communicate. Default bridge doesn't have DNS — use custom bridges.
host
Shares host network stack
No network isolation. Container uses host's IP and ports directly. Best performance, zero isolation.
overlay
Multi-host networking
Used in Docker Swarm. Enables containers on different hosts to communicate as if on the same network.
none / macvlan
none = completely isolated, no networking. macvlan = container gets its own MAC address, appears as physical device on the network.
Network commands
docker network create mynet
docker network ls
docker network inspect mynet
docker network connect mynet web
Custom bridge networks get automatic DNS — containers can reach each other by container name. The default docker0 bridge does NOT have this. Always create named networks for multi-container apps.
06 — Volumes & Storage
Volumes
Docker-managed storage
Stored in /var/lib/docker/volumes. Lifecycle independent of containers. Best for production data persistence.
Bind mounts
Host path mapping
Mounts a specific host path into the container. Great for development (live reload). Risky in production.
tmpfs
In-memory storage
Stored in host memory only. Fast, ephemeral. Good for secrets or sensitive temp data that shouldn't hit disk.
Volume commands
docker volume create mydata
docker run -v mydata:/app/data nginx # named volume
docker run -v $(pwd)/src:/app/src nginx # bind mount
docker run --tmpfs /tmp nginx # tmpfs
docker volume ls && docker volume prune

08 — Registry & Distribution
Registry concepts
Image storage & distribution
Registry stores images. Repository is a collection of related images. Tag identifies a specific version.
Harbor
Enterprise registry
CNCF project. Adds RBAC, vulnerability scanning (Trivy), content trust, replication, and audit logs on top of OCI registry.
OCI manifest
Multi-arch images
A manifest list points to platform-specific images (amd64, arm64). docker buildx builds multi-arch images.
# multi-arch build and push
docker buildx create --use --name multiarch
docker buildx build --platform linux/amd64,linux/arm64 -t registry/app:1.0 --push .
 
# inspect manifest
docker manifest inspect registry/app:1.0
09 — Security
Container security basics
Non-root userNever run as root in containers
Read-only FS--read-only flag where possible
Drop capabilities--cap-drop ALL --cap-add only needed
No privilegedAvoid --privileged in production
Seccomp/AppArmorSyscall filtering profiles
Image scanning
Vulnerability detection
Scan images for CVEs before deployment. Tools: Trivy, Grype, Docker Scout, Snyk. Integrate into CI/CD pipeline.
trivy image app:1.0
docker scout cves app:1.0
Secrets management
Never bake secrets into images. Use build args only for non-sensitive data. Use Docker secrets, env files, or external secret managers (Vault, AWS SM).
Content trust
Docker Content Trust (DCT) uses Notary to sign and verify images. Enable with DOCKER_CONTENT_TRUST=1 to only pull signed images.
10 — Common Interview Questions
Container vs VM
Containers share the host OS kernel via namespaces and cgroups. VMs have a full guest OS. Containers are faster to start, smaller, less isolated.
How do image layers work?
Each Dockerfile instruction creates a read-only layer using a union filesystem (overlayfs). Only the top container layer is writable. Layers are content-addressed by SHA256.
ENTRYPOINT vs CMD
ENTRYPOINT is the fixed executable. CMD provides default arguments. CMD can be overridden at runtime. ENTRYPOINT can be overridden with --entrypoint.
How do containers communicate?
On the same custom bridge network, containers communicate by container name (DNS). On default bridge, by IP only. Host network removes isolation entirely.
How to reduce image size?
Multi-stage builds, minimal base images (alpine, distroless), combine RUN commands, .dockerignore, avoid installing dev tools in final image.
Why does data get lost?
Container's writable layer is destroyed when the container is removed. Use named volumes or bind mounts to persist data outside the container lifecycle.
